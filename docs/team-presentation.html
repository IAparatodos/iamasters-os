1. Contexto: por qué se construyó

Tras analizar las opciones del mercado anglosajón, detectamos un gap: existen sistemas operativos agénticos empaquetados — repos Claude Code listos para clonar e instalar, con skills curadas, brand context, multi-cliente y memoria que aprende — pero ninguno en castellano.

iAmasters NO tenía equivalente propio. Synapsis (el sistema de memoria de Luis Pitik) cubre la capa de aprendizaje, pero no la arquitectura completa: faltan brand context compartido, multi-cliente, skills curadas con colaboración, niveles de proyecto formalizados, etc.

Decisión estratégica de Angel

Construir iAmasters OS con tres ventajas claras frente al resto del mercado:

  • En castellano (el resto solo inglés)
  • Con Synapsis embebido como engine de memoria
  • Como repo público y neutro — sin info privada de Angel ni de la academia. Cualquier operador IA puede clonarlo, instalarlo y usarlo

Resultado: en una sola sesión se construyeron las versiones v0.1.0 → v0.4.0, con esqueleto + 12 skills + 4 templates verticales + scripts de gestión + docs operativos completos.

2. Qué es iAmasters OS

Un repositorio Claude Code listo para clonar e instalar que convierte una sesión vanilla de Claude Code en un sistema operativo profesional para operadores de IA.

Para quién

Operador IA freelance

Sirve a varios clientes desde su laptop. Quiere estandarizar entregables y profesionalizar sin crear estructura corporate.

Empresario / pyme

Automatiza su negocio con Claude Code. Quiere memoria que persista entre sesiones y skills que aprendan de su contexto.

Agencia pequeña (2-10)

Quiere estandarizar el stack agéntico del equipo. Multi-cliente con voice y outputs separados por marca.

Formador / educador

Enseña Claude Code y necesita un repo de referencia que sus alumnos puedan clonar y usar inmediatamente.

Lo que da el primer día

  • Memoria que persiste entre sesiones — no más "explícame tu stack otra vez"
  • 12+ skills curadas instaladas, ya con tu marca dentro
  • Brand context (voz, posicionamiento, ICP) auto-generado en 30 minutos
  • Multi-cliente listo para escalar (4 templates de vertical incluidos)
  • Sistema de aprendizaje continuo: lo que repites se gradúa a regla automática
  • Dashboard visual con métricas reales de uso del operador (heredado de Sinapsis)

4. Arquitectura: 3 capas

iAmasters OS combina tres capas distintas que trabajan juntas pero pueden mantenerse independientes:

Capa 1 · Sinapsis v4.5 (engine de memoria)

vendor/sinapsis/ · vendored intacto · instalado en ~/.claude/

El engine que hace que Claude Code aprenda de ti. Operator state global, instincts auto-aprendidos, skills on-demand, dashboard, dream cycle. Trasplantado íntegro del repo de Luis Pitik sin modificar (preservamos los 112 tests upstream).

Capa 2 · OS (estructura del repo)

brand-context/ · context/ · projects/ · clients/

La estructura que añadimos encima de Sinapsis: brand context (voice, positioning, ICP), agent context (soul, user, learnings), proyectos estructurados, multi-cliente con templates por vertical.

Capa 3 · Skills curadas + commands + scripts

.claude/skills/ · .claude/commands/ · scripts/

12 skills validadas en categorías marketing, operations, strategy, tools, visualization, _meta. 5 slash commands del OS. 4 scripts de gestión (install, add-client, update con conflict resolution, validate-skill).

Decisión clave: vendoring intacto. Sinapsis se mantiene tal cual lo publica Luis. Nosotros no modificamos sus paths ni sus scripts. Cuando Luis publique nueva versión upstream, hacemos git pull en vendor/sinapsis/ y seguimos funcionando. Cero riesgo de romper sus 112 tests.

6. Cómo se instala

Prerequisitos

  • Claude Code CLI instalado y autenticado con plan Pro o Max → claude.ai/code
  • Node.js 18+ → nodejs.org
  • Python 3.9+ (para hooks Sinapsis)
  • Git

Instalación rápida

git clone https://github.com/iamasters-academy/iamasters-os.git
cd iamasters-os
bash scripts/install.sh

Qué hace el instalador (6 pasos)

1

Comprueba prerequisitos

Verifica que Claude Code, Node.js, Python 3 y Git están en PATH. Si falta algo crítico → exit con mensaje claro.

2

Detecta estado de Sinapsis

Si ~/.claude/skills/_catalog.json existe → Sinapsis ya instalado, salta. Si no → continúa al paso 3.

3

Instala Sinapsis (si hace falta)

Ejecuta bash vendor/sinapsis/install.sh que es el installer original de Luis sin modificar. Configura los 7 hooks de Sinapsis en ~/.claude/settings.json (4 PreToolUse + 1 PostToolUse + 1 Stop + 1 PreCompact).

4

Inicializa la capa OS

Crea context/soul.md, context/user.md, context/learnings.md con plantillas. Crea .gitkeep en carpetas vacías para que git las trackee.

5

Instala hook local del OS

Crea scripts/skill-change-detector.sh que detecta cuando se añade/modifica una skill y flagea para sync en próximo wrap-up.

6

Verifica instalación

Comprueba que ~/.claude/skills/ y .claude/settings.json existen. Si todo OK → muestra "siguiente paso".

Tiempo total: ~30 segundos.

7. Primer uso paso a paso (los primeros 30 minutos)

1

Arrancar Claude Code en el repo

cd iamasters-os
claude

Claude lee CLAUDE.md raíz y ejecuta su "MANDATORY first action": leer ~/.claude/skills/_operator-state.json. Como tiene needsOnboarding: true → invoca automáticamente la skill meta-onboarding-wizard.

2

Onboarding wizard (3 minutos)

El wizard hace 7 preguntas:

  1. Avatar: single business / freelance / agencia / formador
  2. Nivel técnico: cero / intermedio / avanzado
  3. Dominio principal: marketing / operations / strategy / dev / mixto
  4. Stack actual (multi-select)
  5. Idioma de outputs cliente (si aplica)
  6. Firecrawl API key (opcional)
  7. Identidad básica: nombre, email, web/LinkedIn

Configura defaults inteligentes y rellena operator-state.json + context/user.md.

3

Brand voice setup (5-10 minutos)

El wizard pregunta si quieres lanzar marketing-brand-voice. Si dices sí:

  • Pegas tus URLs públicas (web, LinkedIn, YouTube, blog)
  • Firecrawl scraping → extrae voice samples
  • 6 preguntas de calibración (tono dominante, vocabulario que evitas, frases-firma, jerga propia, anti-modelo, modelo aspiracional)
  • Genera voice-profile.md + samples.md + 3 registros A/B/C

Tu marca queda "instalada" en el OS. Cualquier skill marketing-* la usará automáticamente.

4

Primera tarea real

Pides algo concreto. Ejemplo: "Escribe 3 versiones de post de LinkedIn sobre [tu tema]".

Claude:

  1. Invoca marketing-copywriting
  2. Lee tu brand-context (voice + positioning + ICP)
  3. Genera 3 variaciones con ángulos distintos (storytelling / contraintuitivo / problema-solución)
  4. Pasa CADA variación por tool-output-verifier (humanizer + brand voice + length + factuality)
  5. Te entrega las 3 con scores y sugerencias
5

Cierre con /wrap-up

/wrap-up

Genera daily summary, sincroniza skills si añadiste alguna, propone Git commit (espera tu OK), te dice qué retomar mañana.

6

Próximos días

Cuando vuelvas a abrir Claude Code en el repo, ejecuta /start-here. Te recapitulará lo de ayer + propondrá tarea del día. La memoria persiste.

11. Los 4 templates verticales

Cada template incluye 6 archivos: README específico al vertical + voice-profile + positioning + ICP (en brand-context/) + soul.md + user.template.md (en context/). Los placeholders {{CLIENT_NAME}} se sustituyen automáticamente al ejecutar add-client.sh.

freelance-ia

Para: operador IA solo, sirve a 3-10 clientes, ticket 5-50K€/proyecto.

Voice: profesional pero accesible (B con toques de C). Confiable, honesto, pragmático.

Positioning sugerido: vertical-specialist o use-case-specialist. Anti-vendehumos.

ICP típico: pymes 5-30 personas con dolor concreto en horas perdidas / clientes en cola / empleado clave que se va.

Skills favoritas: marketing-copywriting, marketing-content-repurposing, tool-output-verifier.

agencia-marketing

Para: equipo 2-10, 5-30 clientes simultáneos, MRR recurrente.

Voice: profesional creativo, equilibrio formal-cercano (B con A para premium). Cada cliente final tiene SU voice.

Positioning sugerido: agencia especialista, no full-service. Foco vertical o servicio.

ICP típico: pymes/scaleups con MRR establecido buscando crecer marketing.

Skills favoritas: marketing-content-repurposing (alta frecuencia output) + marketing-copywriting.

formador-online

Para: coach/educador, vende cursos/comunidad/cohort, ticket 200-2000€.

Voice: cercano, autoridad ganada, vulnerabilidad calibrada (B con C importante).

Positioning sugerido: vertical-first, method-first o anti-genérico. Tu marca personal ES el producto.

ICP típico: persona aspiracional con dolor concreto que puede pagar 200-2000€.

12. Los 4 scripts

scripts/install.sh — installer principal del repo

Delega a vendor/sinapsis/install.sh + inicializa la capa OS. 6 pasos color-coded: prerequisitos → detección Sinapsis → instalación si falta → init capa OS (soul.md, user.md, learnings.md) → hook local → verificación. Tiempo total ~30 segundos.

scripts/add-client.sh — crea cliente desde template

Argumentos: nombre-cliente (kebab-case obligatorio) + template (uno de los 4 verticales o "vacio"). Valida nombre, copia template, sustituye {{CLIENT_NAME}} en todos los .md, genera clients/<nombre>/CLAUDE.md con overrides específicos.

bash scripts/add-client.sh acme-corp freelance-ia

Output: estructura completa creada en clients/acme-corp/, lista para configurar.

scripts/update.sh — pull con conflict resolution

Maneja 4 escenarios:

  1. Sin cambios: ya estás al día, exit limpio
  2. Upstream actualizó, tú no modificaste: pull directo de archivos seguros
  3. Tú modificaste local, upstream no: keep local automáticamente
  4. Conflicto en misma línea: prompt caso-por-caso (k=keep / u=use upstream / d=diff / s=skip)

Backup automático en .backup/<timestamp>/ antes de tocar nada. User data nunca se sobrescribe (brand-context, context, projects, clients, .env).

scripts/validate-skill.sh — valida skill antes de instalar

El motor del comando /install-skill. Pasos:

  1. Parsea URL de GitHub (owner/repo + branch + subpath)
  2. Shallow clone a /tmp/iamasters-os-skill-validate-<hash>/
  3. Valida estructura: SKILL.md presente, YAML frontmatter en líneas 1-3
  4. Valida name kebab-case + description 30-500 chars + verbo de intención
  5. Detecta scripts con patrones peligrosos (rm -rf /, eval, dd if=, mkfs, curl|sh)
  6. Detecta credenciales hardcoded (regex API keys, tokens, passwords)
  7. Comprueba conflicto con skills locales del mismo nombre
  8. Output color-coded: OK · WARN · ERROR + recomendaciones de instalación
scripts/skill-change-detector.sh — hook local

Hook configurado en .claude/settings.json que se dispara en PostToolUse de Edit/Write. Si la edición fue en .claude/skills/, escribe flag en .claude/.skills-pending.json para que el próximo /wrap-up sincronice el catálogo y actualice el registry de CLAUDE.md.

14. Flujo multi-cliente típico

Crear cliente nuevo

bash scripts/add-client.sh acme-corp freelance-ia

Esto crea clients/acme-corp/ con estructura completa: brand-context, context, projects, briefs, CLAUDE.md de overrides.

Trabajar dentro de un cliente

cd clients/acme-corp
claude

Claude Code arranca en el contexto del cliente:

  • Lee CLAUDE.md raíz del repo (instrucciones generales)
  • + clients/acme-corp/CLAUDE.md (overrides específicos)
  • + clients/acme-corp/brand-context/ (voice/positioning/ICP propios del cliente)
  • + clients/acme-corp/context/ (soul/user del cliente)

Ejemplo de día con 3 clientes

# Cliente A (freelance-ia)
cd clients/acme-corp && claude
> "Crea LinkedIn post sobre case study X"
   → marketing-copywriting genera con voice de Acme
> /wrap-up
exit

# Cliente B (agencia-marketing)
cd ../widget-shop && claude
> "Repurpose último video de su CEO en 5 piezas"
   → marketing-content-repurposing
> /wrap-up
exit

# Cliente C (consultoria-b2b)
cd ../north-star-consult && claude
> "Redacta propuesta comercial para nuevo lead"
   → marketing-copywriting registro A formal
> /wrap-up
exit

Best practices

  1. Siempre cd antes de claude. No trabajar en cliente X desde raíz.
  2. Wrap-up al cambiar contexto. Daily summary se separa por cliente.
  3. Brand voice por cliente, no compartir entre clientes.
  4. Una skill custom = un cliente, o promoverla al raíz si aplica a varios.
  5. Backup el cliente regularmente si tiene info crítica (no del git público).

15. Decisiones técnicas clave

Vendoring intacto de Sinapsis

Decidimos NO modificar los paths hardcoded de Sinapsis ($HOME/.claude/skills/...) en vendor/sinapsis/. Razones:

  • Sinapsis tiene 112 tests pasando con esos paths. Modificarlos rompe upstream.
  • Cuando Luis publique nueva versión, hacemos git pull en vendor y seguimos funcionando.
  • El installer de Sinapsis está pensado para instalación global. Forzarlo a in-repo era mucho trabajo de reescritura sin valor real.

Implicación: Sinapsis vive en ~/.claude/ (global del operador). El repo iAmasters OS solo añade capas encima. Cross-project memory de Sinapsis aplica a todos los repos del operador, no solo a iAmasters OS.

Repo público y neutro (sin info privada)

Decisión explícita de Angel: el repo es público (privado en v0 pero pensado para abrirlo) y neutro. Cualquier operador puede clonarlo sin que se cuele info de Angel ni de iAmasters academy.

Implicación: los templates de cliente están en blanco con placeholders {{CLIENT_NAME}}. Los ejemplos en docs son genéricos. Las skills no asumen ningún avatar concreto.

Sinapsis embedded como engine, no como capa secundaria

Los competidores del mercado anglosajón tienen un learnings.md plano por skill. Eso es un primer paso muy básico. Synapsis ya está varios niveles por encima: instincts con confidence score, evolución a skills/reglas, projects registry estructurado, dream cycle de hygiene, dashboard.

Decisión: trasplantamos Sinapsis íntegro como engine principal, no como capa secundaria. Eso es nuestra mayor ventaja competitiva.

Patrón canónico de skill definido y enforced

El meta-skill-creator define las reglas no-negociables para skills nuevas:

  • YAML frontmatter completo (name kebab-case + description 50-500 chars con verbo de intención)
  • Progressive disclosure (knowledge extenso en references/)
  • Steps numerados y testables
  • Skill collaboration explícita
  • Output verifier gate si genera entregables
  • Learnings hook al cierre

Implicación: cualquier skill nueva (interna o instalada via /install-skill) debe cumplir el patrón. scripts/validate-skill.sh hace check automático.

3 registros de voice (A/B/C) en lugar de 1 voice profile genérico

El estándar habitual es 1 voice profile genérico. Pero la realidad operativa es que escribimos muy distinto en email premium vs LinkedIn vs WhatsApp. Si el OS no lo modela, todos los outputs salen homogéneos.

  • A formal: email cliente premium, propuesta comercial, contrato
  • B divulgativo: LinkedIn post, blog, video script, newsletter (95% del contenido público)
  • C cercano: WhatsApp comunidad, DM, comentarios redes

El marketing-copywriting auto-detecta el registro correcto según plataforma + purpose.

tool-output-verifier como gate obligatorio

Todas las skills marketing-* invocan tool-output-verifier en su último paso. No es opcional.

Por qué: en operación real, los operadores se saltan los gates "porque tengo prisa" y entregan AI slop al cliente. Hacerlo obligatorio fuerza el patrón profesional.

Threshold por plataforma: humanizer ≥8 para email premium · ≥7 para LinkedIn/blog · ≥6 para WhatsApp comunidad. Brand voice match siempre ≥7.

Backups automáticos en update.sh

Antes de aplicar cualquier update, scripts/update.sh crea backup automático en .backup/<timestamp>/ con: .claude/skills/, brand-context/, context/, projects/, clients/, .env, .claude/settings.json.

Política: nunca confiar en que algo "no se va a tocar". Backup siempre, recovery 1-comando.

User data nunca en git público

.gitignore extendido excluye:

  • brand-context/voice/voice-profile.md (y samples, registers)
  • brand-context/positioning/positioning.md
  • brand-context/icp/icp.md
  • brand-context/assets/
  • context/user.md y context/daily-memory/
  • projects/briefs/* y outputs por skill
  • clients/* excepto _templates/
  • .env

Solo se commitean: README, soul.md, learnings.md (genérico), templates vacíos.

Validate-skill no elimina TMP automáticamente

Cuando alguien intenta instalar una skill externa con /install-skill, el script valida pero NO elimina el TMP dir. Razones:

  • El operador puede inspeccionar manualmente la skill descargada
  • Si validación falla, podría haber falso positivo y querer revisar
  • Deja al operador decidir cuándo limpiar (rm -rf /tmp/iamasters-os-skill-validate-*)
Curated lists son opinated, no inclusivas

Las listas docs/mcps-curated.md y docs/skills-recommended.md NO son catálogos exhaustivos. Solo incluyen lo que validamos en producción ≥2 semanas. Razones:

  • Inclusión sin filtro causa decisión paralysis
  • El valor real de una curated list es el filtro
  • "Si no sé, no lo recomiendo" es mejor que "lo metemos por si acaso"

Para contribuir a las listas hace falta experiencia real documentada (ver sección "Cómo contribuir" en cada doc).
